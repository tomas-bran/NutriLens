# NutriLens — Auditoría de código (2026-05-18)

Auditoría puntual antes de la próxima iteración. Objetivo doble:

1. Detectar **falencias generales** que valga la pena resolver antes de seguir agregando features.
2. Diseñar un **plug-n-play** real para el proveedor de IA, anticipando dos escenarios: que se renueve el acceso a Azure Foundry, o que tengamos que cambiar a otro proveedor (Azure OpenAI, OpenAI directo, Anthropic, …).

Las propuestas vienen agrupadas por severidad y con un PR sugerido para cada una. Nada se ejecuta sin tu aprobación.

---

## 1. Plug-n-play del proveedor de IA

### 1.1 Estado actual

- Selector único en `src/lib/ai/index.ts`: lee `process.env.IA_PROVIDER` y construye `MockIaProvider`, `FoundryProvider`, o **falla silenciosamente a Mock** si pide `azure-openai` (no implementado) o cualquier valor desconocido.
- `FoundryProvider` está acoplado al SDK `openai` apuntando al endpoint `/openai/v1` de Azure Foundry. Reusa el mismo cliente para multimodal + mini-text.
- Variables requeridas hardcodeadas: `AZURE_AI_FOUNDRY_ENDPOINT`, `AZURE_AI_FOUNDRY_KEY`, `AZURE_AI_FOUNDRY_MODEL_MULTIMODAL`, `AZURE_AI_FOUNDRY_MODEL_MINI`. Si falta cualquiera, el constructor tira en startup.
- `MockIaProvider` devuelve un único producto fijo: cualquier imagen analizada termina como “Mock Product” categoría `otros`. Sirve para tests pero no para demo manual.

### 1.2 Problemas

| #   | Severidad | Falencia                                                                                                                                                                                                             |
| --- | --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P1  | **Alta**  | Si Azure Foundry se cae o expira el acceso, **no hay segundo proveedor**. El flag `azure-openai` está documentado en `.env.example` pero el switch lo redirige a Mock sin avisarle al usuario final (solo log warn). |
| P2  | **Alta**  | Las env vars son **provider-specific** (`AZURE_AI_FOUNDRY_*`) en el código del provider. Agregar OpenAI directo o Anthropic exige duplicar `required('XXX')` calls y vivir con todas las vars siempre seteadas.      |
| P3  | Media     | El cliente `OpenAI` es construido en el constructor; no hay forma de inyectar credenciales por request (necesario si queremos rotar keys o usar credenciales por tenant).                                            |
| P4  | Media     | `MockIaProvider` siempre devuelve el mismo producto. Para demos manuales (sin Azure) es ciego — no podemos probar “riesgo alto” vs “bajo”.                                                                           |
| P5  | Baja      | El selector cachea singleton, no expone reset público (sí `_resetIaProvider`). Cambiar `IA_PROVIDER` en runtime requiere restart.                                                                                    |

### 1.3 Propuesta — Refactor en 3 PRs

#### PR-A: Config-driven provider (1-2 días)

Centralizar la config en un objeto tipado:

```ts
// src/lib/ai/config.ts
export interface IaProviderConfig {
  kind: 'mock' | 'foundry' | 'azure-openai' | 'openai';
  endpoint?: string;
  apiKey?: string;
  multimodalModel?: string;
  miniModel?: string;
  timeoutOverrides?: Partial<{ analyze: number; explain: number; chat: number }>;
}

export function loadIaConfig(env: NodeJS.ProcessEnv = process.env): IaProviderConfig { ... }
```

Cambios:

- `getIaProvider()` deja de leer `process.env.*` directo; recibe el config.
- `FoundryProvider`/`AzureOpenAIProvider`/`OpenAIProvider` reciben `{ endpoint, apiKey, models }` por constructor, no leen env.
- Errores de config se concentran en `loadIaConfig` con mensajes claros: `"Falta AZURE_OPENAI_KEY: configurá la key o cambiá IA_PROVIDER"`.
- Test: `loadIaConfig` con tabla de inputs válidos/inválidos.

Beneficio: agregar un proveedor nuevo = 1 archivo `<name>-provider.ts` que implementa `IaProvider` + 1 case en `loadIaConfig`. Sin tocar el resto.

#### PR-B: OpenAIProvider (provider de respaldo, 2 días)

Implementar `OpenAIProvider` que pega contra `api.openai.com` con `OPENAI_API_KEY`. Mismo SDK `openai`, mismo `callChat` que ya tenemos. Modelos por env:

```
OPENAI_API_KEY=sk-...
OPENAI_MODEL_MULTIMODAL=gpt-4o-mini      # o gpt-4o
OPENAI_MODEL_MINI=gpt-4o-mini
```

Tests:

- Unit con SDK mockeado (igual que `foundry-provider.test.ts`).
- Integration opcional con OPENAI key real, marcada `@live`, skippeada por default.

#### PR-C: Mock dataset (4-6 horas)

Reemplazar el `FIXED_EXTRACTION` por un dataset de 5-10 respuestas “realistas” seleccionadas por **hash determinista del buffer**:

```ts
const FIXTURES: ProductExtraction[] = [ /* galletitas alto, snack bajo, lácteo medio, ... */ ];
analyzeLabel(buffer) → FIXTURES[hash(buffer) % FIXTURES.length];
```

Permite demos manuales sin Azure: cada imagen distinta da una respuesta distinta y reproducible.

### 1.4 Fallback automático (opcional, PR-D)

Si querés que cuando Foundry tire 5xx persistente caigamos a `OpenAIProvider` sin restart:

```ts
class FailoverProvider implements IaProvider {
  constructor(private primary: IaProvider, private fallback: IaProvider) {}
  async analyzeLabel(...) {
    try { return await this.primary.analyzeLabel(...); }
    catch (e) {
      if (isPersistent5xx(e)) return this.fallback.analyzeLabel(...);
      throw e;
    }
  }
}
```

Configurable: `IA_PROVIDER=foundry+openai` arma `FailoverProvider(foundry, openai)`. Tiene costo (key de los dos vendors), por eso lo dejo opcional.

### 2.2 Anti-patterns / oportunidades

- **No hay rate limiting** en los endpoints públicos (`/api/analyze`, `/api/chat`). Un usuario podría llamar `analyze` infinitas veces y consumirnos el budget de IA. Sugerencia: middleware `rateLimit({ window: '1m', max: 10 })` con tabla en memoria (suficiente para MVP) o `next-rate-limit`.
- **No hay autenticación**. Demo abierta. Si va a producción/demo público, agregar al menos un `X-API-Key` chequeable.
- **`/api/admin/timings`** (E06 §9.2) está en spec pero no implementado. No es crítico.
- **Coverage del pipeline E2E**: cubrimos cada step individualmente pero no la cadena completa con un fixture “bad path” (extracción inválida → corrective retry → falla persistente). Vale la pena un test integración largo.
- **Telemetría**: los `logger.info` quedan en stdout. Para el demo deployado conviene shipearlos a un sink (Sentry, Datadog free tier, Vercel logs).

### 2.3 Cosas que están bien y conviene mantener

- Arquitectura de steps del pipeline (`validate_file → extract → apply_rules → compute_risk → explain → persist`) — bien abstraída, fácil de testear paso a paso.
- Schemas Zod canónicos en `src/packages/schemas/` con enums en español → mantiene consistencia entre UI y modelo.
- `IaProvider` interface con `MockIaProvider` → ya está la abstracción correcta, solo falta más implementaciones.
- 50 productos seed con UUIDs estables (US-38) → permite tests E2E robustos.
- `LocalStorage` con `existsSync` idempotente → no reescribe el mismo hash dos veces.

## 3. Persistencia de imágenes (preparar para cloud)

### 3.1 Estado actual

- Cada análisis guarda el archivo subido en `./uploads/<sha256>.<ext>` (gitignored).
- La DB guarda **solo el `imagenPath`** (URL relativo). No guarda `mime`, `bytes` ni metadata.
- Si reseteamos `uploads/` perdemos las imágenes; la DB queda con paths rotos.
- `AZURE_BLOB_CONNECTION_STRING` documentado pero sin implementación.

### 3.2 Propuesta — PR-E (medio día)

1. Extender `Product` con `imagenMime: String`, `imagenBytes: Int`, `imagenSha256: String` (ya tenemos hash pero solo como `fileHash`). Migración Prisma.
2. `LocalStorage.save()` ya guarda; cambiar para devolver `{ path, mime, bytes }` en vez de solo `path`.
3. `persist` step popula los 3 campos nuevos.
4. UI no cambia: sigue mostrando `imagenPath` como ahora.
5. `AzureBlobStorage` queda como TODO (implementable en una iteración futura sin cambios en la API del Storage).

Esto te deja la data lista para cuando deployes: las imágenes están en disco con metadata persistida, y solo cambia el adapter cuando muevas a S3/Blob.
